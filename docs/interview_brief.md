# 面试讲解提纲

## 一句话介绍

这是一个中文内容审核策略原型，不只是文本分类器。  
它把模型识别、规则兜底和审核动作串成了一条完整的审核决策链路。

## 建议先讲的三点

### 1. 为什么不是只做分类

- 内容风控真正关心的是处置动作，不是单个标签本身
- 所以项目输出的不只是 `label`，还包括 `risk_score`、`risk_band` 和 `action`

### 2. 为什么要保留人审

- 真实风控不是只有“放行”和“拦截”
- 很多样本存在模糊表达、轻度暗示或证据不足的情况
- 这类内容更适合进入 `review`

### 3. 为什么是规则和模型并存

- 模型负责语义识别，处理变体表达
- 规则负责兜底高确定性模式，例如联系方式、返利、代理、私聊诱导
- 规则主要负责升级审核动作，而不是和模型做严格概率融合
- 两者结合，比纯规则或纯模型更接近真实业务

## 如果被问到“规则和模型怎么结合”

推荐直接这样回答：

- 模型先给出风险类型判断
- 规则只负责补充高确定性证据，并升级审核动作
- 强规则直接拦截，中强规则进入人审，其余由模型风险分和置信度决定

不要把它讲成严格的概率融合，更合适的说法是：

- 这是一个启发式审核策略原型
- 目标是可解释、可演示、可落地，而不是追求数学上的最优融合

## 为什么选 MacBERT

- `hfl/chinese-macbert-base` 是成熟的中文 base 模型
- 相比更大的模型，本地微调成本更低
- 对 `RTX 4060 8GB` 这类机器更友好，适合在半小时内完成训练和演示准备

## 数据怎么讲

- 主数据集使用 `ChineseHarm-Bench`，将 `不违规 / 谩骂引战 / 低俗色情 / 黑产广告` 映射到当前四类标签
- `abuse` 在需要时可用 `COLDataset` offensive 样本做补强
- `normal / sexual / ad` 使用定向构造样本
- 重点不是追 benchmark，而是把通用模型调成更贴近审核场景的原型

## 项目局限

- 当前数据是展示级数据，不是生产级数据
- 标签体系还比较粗
- 目前只覆盖文本，不覆盖图片和视频
- 阈值策略仍然需要线上反馈继续校准

## 如果面试官继续追问

可以往这几个方向展开：

- 如何降低误杀和漏放
- 如何做误判样本回流
- 如何扩充广告变体和绕过表达
- 如何把展示页升级成审核工作台
